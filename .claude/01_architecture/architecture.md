# アーキテクチャ設計書

## 概要

このドキュメントは、秩父三十四箇所巡礼アプリ（Expo / React Native版）の全体アーキテクチャと設計方針を定義します。

## アーキテクチャスタイル

### 基本方針

- **シンプルなクライアントサイドアーキテクチャ**
  - バックエンドAPIは持たない（外部APIのみ使用）
  - 状態管理は最小限に抑える
  - React Native のネイティブ機能を活用

### レイヤー構成

```
┌─────────────────────────────────────┐
│       Presentation Layer            │
│  (Expo Router Screens / Components) │
├─────────────────────────────────────┤
│       Business Logic Layer          │
│  (Custom Hooks / Utils)             │
├─────────────────────────────────────┤
│       Data Access Layer             │
│  (Static Data / AsyncStorage)       │
├─────────────────────────────────────┤
│       External Services             │
│  (react-native-maps / OSRM)         │
└─────────────────────────────────────┘
```

## 各レイヤーの責務

### 1. Presentation Layer（プレゼンテーション層）

**責務**: UIの表示とユーザー操作の受付

**構成要素**:
- **Screens**: Expo Router のスクリーンコンポーネント（ルーティング）
- **Components**: 再利用可能なUIコンポーネント
- **Layouts**: タブレイアウトや共通レイアウト

**原則**:
- ビジネスロジックを含めない
- 表示とイベントハンドリングのみ
- 可能な限りステートレスに保つ

### 2. Business Logic Layer（ビジネスロジック層）

**責務**: アプリケーション固有のロジックと状態管理

**構成要素**:
- **Custom Hooks**: 状態管理とロジックのカプセル化
- **Utils**: 汎用的なユーティリティ関数
- **Types**: TypeScript型定義

**原則**:
- カスタムフックで状態とロジックを分離
- 純粋関数を優先
- テスト可能な設計

### 3. Data Access Layer（データアクセス層）

**責務**: データの取得・保存・更新

**構成要素**:
- **Static Data**: 札所情報（JSON）
- **AsyncStorage**: 訪問記録の永続化
- **API Client**: 外部API呼び出し

**原則**:
- データソースの抽象化
- エラーハンドリングの統一
- キャッシュ戦略の明確化

### 4. External Services（外部サービス層）

**責務**: 外部ライブラリ・APIとの連携

**構成要素**:
- **react-native-maps**: 地図表示（iOS: Apple Maps / Android: Google Maps）
- **OSRM (Open Source Routing Machine)**: 経路検索
- **expo-location**: 現在位置取得

## 状態管理戦略

### 基本方針

- **グローバル状態管理ライブラリは使用しない**（Redux、Zustand等）
- React標準のフックで十分な規模

### 状態の種類と管理方法

| 状態の種類 | 管理方法 | 例 |
|-----------|---------|-----|
| サーバー状態 | なし（静的データのみ） | 札所情報 |
| クライアント状態 | useState / useReducer | 地図の表示状態、選択中の札所 |
| 永続化状態 | AsyncStorage + Custom Hook | 訪問記録 |
| URL状態 | Expo Router params | 選択中の札所ID |

### カスタムフック設計

```typescript
// 例: 訪問記録管理
useVisitedTemples()
  → AsyncStorage から読み取り
  → 訪問/未訪問の切り替え機能
  → 進捗率計算

// 例: 現在位置管理
useUserLocation()
  → expo-location から取得
  → パーミッションリクエスト
  → エラーハンドリング
  → 位置情報の更新

// 例: 経路検索
useRouting(from, to, mode)
  → OSRM API 呼び出し
  → ローディング状態管理
  → エラーハンドリング
```

## ルーティング設計

### ページ構成（Expo Router）

```
app/
├── _layout.tsx              # ルートレイアウト
├── (tabs)/
│   ├── _layout.tsx          # タブレイアウト
│   ├── index.tsx            # 地図タブ
│   ├── temples.tsx          # 札所一覧タブ
│   └── progress.tsx         # 進捗タブ
├── temple/
│   └── [id].tsx             # 札所詳細画面
└── modal.tsx                # モーダル
```

### ナビゲーション方針

- **ボトムタブナビゲーション**: モバイルファーストのため、下部に固定
- **タブ形式**: 地図、一覧、進捗の3つの主要画面
- **スタックナビゲーション**: 詳細画面への遷移

## パフォーマンス戦略

### 最適化方針

1. **React.memo / useMemo の活用**
   - 34箇所のマーカー描画を最適化
   - 重い計算結果をメモ化

2. **FlatList の使用**
   - 札所一覧は `FlatList` で仮想化
   - 初期表示の高速化

3. **画像最適化**
   - `expo-image` コンポーネント使用
   - キャッシュ戦略を活用

4. **AsyncStorage キャッシュ**
   - 訪問記録は初回読み込み後メモリにキャッシュ

## セキュリティ方針

### 基本原則

- **個人情報は扱わない**: 訪問記録のみ（デバイスローカル）
- **位置情報のパーミッション**: expo-location でユーザー許可を適切に取得
- **APIキーの保護**: 環境変数（.env）で管理、git管理外

### 環境変数

```
# OSRMは完全無料でAPIキー不要のため、環境変数は不要
```

## エラーハンドリング戦略

### エラーの種類と対応

| エラー種別 | 対応方法 |
|-----------|---------|
| 位置情報取得失敗 | デフォルト位置（秩父市中心）に設定 + エラー表示 |
| 位置情報パーミッション拒否 | パーミッション案内ダイアログを表示 |
| 経路検索失敗 | エラーメッセージ表示 + リトライボタン |
| AsyncStorage 失敗 | メモリ上で動作継続 + 警告表示 |
| 地図読み込み失敗 | フォールバックUI表示 |

## テスト戦略

### 最小限のテスト方針

このプロジェクトは個人開発・小規模なため、包括的なテストは不要。
重要な部分のみテストする。

### テスト対象

| 対象 | テスト手法 |
|------|----------|
| ビジネスロジック（Custom Hooks） | Jest + React Native Testing Library |
| ユーティリティ関数 | Jest 単体テスト |
| UI コンポーネント | 手動テスト |
| E2E | 手動テスト（Detox 不要） |

## デプロイ戦略

### EAS デプロイ

- **プレビュー**: EAS Update で OTA アップデート
- **本番環境**: EAS Build で iOS/Android ビルド
- **環境変数**: `eas.json` で環境別設定

### ビルド設定

```json
{
  "scripts": {
    "start": "expo start",
    "android": "expo start --android",
    "ios": "expo start --ios",
    "lint": "expo lint"
  }
}
```

## まとめ

- **シンプルさ最優先**: 必要最小限の構成
- **クライアント完結**: サーバーレス・状態管理ライブラリ不要
- **モバイルファースト**: React Native のネイティブ機能を最大活用
- **拡張可能**: 将来的な機能追加にも対応可能な設計
